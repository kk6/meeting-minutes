# ADR-0003: Vocabulary Hints for Transcription and Summarization

## Status

accepted

## Date

2026-04-29

## Context (Why)

リアルタイム文字起こしと議事録生成では、固有名詞・人名・社内用語・略語の誤認識が品質を最も損なう要素になっている。既定の音声認識モデルは一般語彙に最適化されており、会議固有の語彙（参加者名、プロジェクトコード、製品名、業界用語）には弱い。

結果として：

- 人名の取り違えが TODO の担当者誤割当てにつながる
- 社内略語が一般語に誤変換され、検索性・可読性が落ちる
- 要約段階では文字起こしの誤りを正規化する手がかりがなく、表記揺れがそのまま残る

会議ごとに既知の語彙（参加者一覧、議題関連の用語集）を事前に提示できれば、文字起こしのバイアスとして、また要約での表記正規化の手がかりとして活用できる。

## Intent (What)

ユーザーが会議開始前に**用語集**と**参加者一覧**をファイルとして配置することで、文字起こしと議事録生成の双方が、その語彙を「期待される表現」として認識・尊重する。

観察可能な振る舞い：

- 用語集に登録した固有名詞・専門用語が、文字起こし結果でその表記のまま現れやすくなる
- 参加者一覧に登録した人名・役職が、議事録の「担当者」「発言者」欄で正しい表記で扱われる
- ファイル未配置でも従来通り動作する（ヒント機能はオプショナル）
- 設定ファイルから語彙ファイルの所在を指定できる

入出力の契約：

- 入力: テキストファイル（行区切りで語彙を列挙、コメント可）
- 効果範囲: その実行セッション全体（チャンク単位で一貫して適用される）
- 失敗時挙動: ファイル不在・読み込みエラーは警告のみで処理継続（精度向上は best-effort）

## Constraints

- **語彙ヒントの容量制限**: 文字起こしモデルが受け付ける事前ヒントには上限があるため、全件投入が不可能な場合は頻出語を優先する戦略が必要
- **後方互換性**: 既存の `config.toml` を変更せずに動作すること（新セクションはオプショナル）
- **機微情報の扱い**: 用語ファイル自体は VCS にコミットさせない設計とする（`.gitignore` ガイダンスを README/サンプルで提示）
- **保証ではなくバイアス**: ヒント投入は確率分布の調整であり、誤認識ゼロを保証しない

## Alternatives Considered

1. **要約 LLM 側だけで表記揺れを補正する**
   却下理由: 文字起こし時点で別語に化けた固有名詞は、後段の LLM では元語彙に戻せないことがある。上流で抑制するほうが効果的。

2. **会議ごとにモデルを fine-tuning する**
   却下理由: 運用コストが過大。会議単位で軽量に切り替える要件と合わない。

3. **語彙をハードコードまたは単一グローバルファイルに集約**
   却下理由: 会議ごとに参加者・話題が変わる前提に合わない。会議単位で差し替え可能な配置が望ましい。

## Consequences

ポジティブ：

- 固有名詞・人名の誤認識が体感レベルで減る
- 議事録の表記揺れが減り、後工程（検索、共有、引用）の負担が下がる
- 会議準備の一部として「用語集・参加者リストを用意する」運用が定着すれば、品質と準備の双方が改善する好循環が生まれる

ネガティブ／トレードオフ：

- 会議ごとにファイルを用意する手間が発生する（オプショナルなので強制ではない）
- 不適切に大量の語彙を投入するとモデル性能を損なう可能性がある → 容量超過時の挙動（切り詰め・優先順位）を明示する必要がある
- 用語ファイルが機微情報を含む可能性があり、誤コミット防止のガイダンスが必要
