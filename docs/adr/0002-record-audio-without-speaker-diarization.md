# ADR-0002: Record Audio Without Speaker Diarization

## Status

proposed

## Date

2026-04-29

## Context (Why)

ADR-0001 では、リアルタイム話者推定の精度不足を受けて、録音済み音声全体を使った後処理で話者ラベルを補正する方針を検討した。

その後、後処理型の話者推定も試したが、同一人物が途中で別 speaker として分裂しやすく、話者切替境界も議事録生成に使えるほど安定しなかった。既知の話者数を指定しても、短い話者断片の smoothing を行っても、十分な改善には至らなかった。

さらに、音声から単語時刻付き transcript を再構築して話者境界で切り直す案も試したが、日本語の文字起こし品質が低下し、議事録本文としては既存の transcript より不自然になった。

一方で、会議主催者側で録画・録音が提供されない場合でも、ローカルに音声だけ残っていれば、後から内容確認や再処理ができる。録音保存は、現在の話者推定精度に依存せず価値がある。

## Intent (What)

現時点では、話者推定を `draft` / `finalize` の議事録生成フローには組み込まない。

`live` セッションでは、文字起こしと並行して会議音声をローカルに保存する。保存された音声は、会議後の内容確認、手動確認、または将来のより高精度な話者推定・再処理の入力として利用できる。

議事録生成は、話者ラベルよりも transcript の日本語品質、決定事項、TODO、未決事項の抽出を優先する。

## Constraints

音声と transcript はローカル環境内で処理し、クラウド API への送信を前提にしない。

録音保存はライブ中の文字起こし体験を損なわないことを優先する。録音のために音声入力の drop や大きな表示遅延を発生させない。

利用者が録音保存を望まない場合に無効化できること。

将来、話者推定ライブラリや録音条件の改善によって再挑戦できるよう、録音済み音声と transcript の時刻対応を残す。

## Alternatives Considered

後処理 diarization を `draft` / `finalize` に組み込む案を検討した。実際に試したところ、同一人物の speaker 分裂や短い誤断片が多く、議事録に話者ラベルを混ぜると品質が下がるリスクが高かった。

音声から単語時刻付きで transcript を再構築し、話者境界で切り直す案を検討した。話者境界に沿った分割は可能だったが、日本語の文字起こし品質が落ち、議事録本文の入力としては不適切だった。

録音保存自体を見送る案も考えられるが、会議後に音声が残る価値は話者推定とは独立して高い。特に、外部会議で録画・録音が提供されない場合の保険として有用である。

## Consequences

現時点の実装は、話者推定の不安定さを議事録品質へ持ち込まない。

会議音声は `audio_live.wav` としてローカルに残り、後から確認・再処理できる。

話者別の議事録や自動 speaker ラベルは提供しない。

将来、リアルタイムまたは後処理の話者推定精度が改善した場合、保存済み音声を使って再評価できる。
