# ADR-0001: Post-process Speaker Diarization

## Status

proposed

## Date

2026-04-29

## Context (Why)

リアルタイム文字起こしでは、Whisper medium による segment 表示は精度と反応速度のバランスが良く、会議中に内容を追う用途では十分実用的だった。

一方で、リアルタイム話者推定は pyannote provider と diart provider の両方を試したが、話者切替境界や複数話者の一貫した識別で十分な精度に達しなかった。特に、低遅延で処理するための chunk 単位推論では、文字起こし segment、音声 chunk、話者推定区間の境界がずれやすく、ライブ表示中のラベルが不安定になる。

会議中の体験を悪化させず、議事録生成時の品質を上げるには、リアルタイム中の話者推定に過度な責務を持たせるより、録音済み音声全体を使った後処理で話者ラベルを補正する方が現実的である。

## Intent (What)

ライブ中は文字起こしの反応速度と読みやすさを優先し、Whisper segment 単位の transcript 表示を安定版の中心機能とする。

話者ラベルは、ライブ中の補助表示ではなく、会議終了後に録音済み音声と transcript を突き合わせて補正できる後処理機能として扱う。

後処理後の transcript は、各発話 segment に対してより一貫した話者ラベルを持ち、議事録ドラフトや最終議事録生成の入力として利用できる。

## Constraints

音声と transcript はローカル環境内で処理する。

ライブ中の文字起こし体験を損なわないことを優先する。話者推定のために音声入力の drop や大きな表示遅延を発生させない。

話者推定は実名識別ではなく、同一話者らしい発話を `Speaker 1`、`Speaker 2` のように追跡しやすくする補助情報として扱う。

既存の realtime transcript 形式と minutes 生成フローとの互換性を保つ。

## Alternatives Considered

リアルタイム pyannote provider を改善し続ける案を検討した。音声履歴や更新間隔を調整することで drop は抑えられたが、話者切替境界の精度とラベル一貫性には限界が残った。

diart provider を導入してオンライン話者推定を試す案を検討した。pyannote とは異なる挙動になったが、今回の用途で明確に精度が改善したとは言えなかった。また、依存関係やモデル互換性の扱いも複雑になった。

リアルタイム中に Whisper medium と高精度 diarization を同時に実行する案を検討した。ローカル実行では処理負荷が高く、会議中の反応速度と安定性を損なうリスクが高い。

## Consequences

ライブ中の表示は軽く安定し、文字起こしを追う体験を優先できる。

話者ラベルの品質改善は、リアルタイム性ではなく後処理の品質として扱えるため、録音全体を参照した一貫性のある推定を試しやすくなる。

会議終了後に補正処理を実行する追加ステップが必要になる。

リアルタイム話者ラベルは安定版機能としては扱わない。
